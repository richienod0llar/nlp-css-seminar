The Problem

Writing good survey questions is hard, manual, expert work

  • Social science runs on surveys, but a bad question gives invalid data.
  • Researchers start from a vague indicator (e.g. "satisfaction with care") and must turn it into a precise, neutral, theory-grounded question.
  • This requires applying a strict measurement framework consistently across hundreds of items: slow and expensive.

Our research question

Can an open LLM, guided only by prompts, reproduce this expert process: picking the right measurement category and writing the question?

If yes, it accelerates questionnaire design and makes it more consistent and reproducible.

The core motivation: questionnaire design is a bottleneck. It's skilled, slow work. We test whether an LLM can do it following the same rules a trained annotator would.

3 Semantic Structures

The grammar of the assertion, written as short codes: xIe, rDy, xPRy … Each concept allows only certain codes.

Think of it as grammatical parsing for survey items: first classify the item, then build it to a template.

This is the annotation guide (provided). Two things must be tagged: the concept (the "what") and the semantic structure (the "grammar", as a code). The codes look cryptic but each letter has a meaning: x = subject, I = link verb, e = evaluative adjective, so xIe is roughly "I am [good/satisfied]".

Why the middle step matters

The assertion + tags force the model to commit to a measurement decision before it writes pretty wording. That decision is where the difficulty, and most errors, live.

A complete trace. The assertion stage is the "thinking": classify and template. The question stage is the "writing". We evaluate them separately.

Assertion Developer

The hard, theory-laden classification + templating step. Scored on concept and structure accuracy.

Question Developer

Turns an assertion into a respondent-facing question. Scored on coverage and wording match.

Both implemented as the same Qwen3.5-9B behind two carefully written system prompts (the rulebook is injected from concepts.yaml).

Two prompts, one model. The concept rulebook is injected programmatically into the prompt so we never hand-edit rules: change the YAML, the prompt updates.

Design decisions: the "why"

Why these tools? (1/2)

ChoiceWhy we made it
Qwen3.5-9B
(open model)		Strong open model that fits on one GPU. Runs locally on LRZ → reproducible, free, and no survey data leaves the cluster (privacy). No vendor lock-in.
Prompting
before fine-tuning		Only 115 examples, too few to train reliably. Prompting is cheap, fast to iterate, and interpretable: we can read exactly why a choice was made. It sets an honest baseline before any training.
vLLM servingFast inference with an OpenAI-compatible API → our pipeline is a simple Python client. Easy to swap models later.
Anticipating the obvious questions. Why not GPT-4 or Claude? Reproducibility, cost, privacy, and the requirement to run on the university cluster. Why not fine-tune immediately? We don't have the data yet, and we want to know how far prompting alone gets us.
Design decisions: the "why"

Why these tools? (2/2)

ChoiceWhy we made it
Structured JSON output
(guided decoding + concept enum)		Guarantees machine-parseable output and forces the model to pick a valid concept from the 22: it cannot invent categories.
enable_thinking = falseQwen3.5 otherwise emits long reasoning prose that breaks JSON parsing. Disabling it gave reliable structured answers.
Isolated evaluationScore each agent independently: the Question Developer is fed the gold assertion, not the model's own. Errors don't compound, so we can localise failures.
Normalization layerFair scoring: unify Behavior=Behaviour, extract the structure code from messy text, without ever editing the gold file.
These four engineering decisions are what made the evaluation trustworthy. Isolated evaluation especially: if we chained the agents, a bad assertion would unfairly tank the question score and we couldn't tell which stage failed.
How we measure success

Evaluation metrics

Assertion stage

• Concept accuracy: right category?

• Structure accuracy: right code?

• Both correct: the strict bar

Question stage

• Coverage: produced a non-empty question?

• Exact match vs gold wording

• Question format classified

One thing to keep in mind

Exact string match understates quality. "How satisfied are you with this product?" vs "…with your product?" are both perfect, but don't match exactly.

→ Run 4 adds an LLM-as-judge alignment score (1–5) that measures meaning, not strings (next section).

All metrics are exact-match after normalization: objective and cheap. The honest limitation: exact match is brutal on the question stage because many phrasings are equally valid. That's why we added the judge metric in Run 4.

Key Findings: by category

Some concepts are easy, some are hard

Near-perfect

Evaluation, Demographics, Preference, Importance, Knowledge, Norms, Rights

The pattern

Frequent, well-defined concepts are easy; rare, abstract ones (Policies, Place, Values, Feelings) are hard.

Clear pattern: the model nails common, concrete concepts. It struggles with rare or abstract ones: partly genuine difficulty, partly tiny n=3 samples where each error costs 33 points.
Key Findings: the core insight

Structure codes are the bottleneck

  • Points below the diagonal = concept right, but structure wrong.
  • Whole concepts get the category right yet score 0% on structure: Expectations of future events, Action tendencies, Feelings, Procedures.
  • Structure codes are fine-grained and theory-laden: e.g. intention (rFDy) vs future event (xFD), distinctions even experts debate.

Takeaway

Identifying what is measured is largely learnable by prompting. Pinning the exact grammar code is the remaining hard problem.

This is the single most important finding. Concept ID responds well to prompting; structure assignment lags. The gap is concentrated in a few concepts where the model knows the category but picks an adjacent structure code.
Key Findings: error analysis

Where it goes wrong (and why it's reasonable)

Cognitive judgment→Events, Place/Quantities/Time→Demographics, Policies→Norms · structure: xFD→rFDy (future event vs intention), xPRy↔xIpr. These are semantically adjacent: the model lands one cell away, not wildly off.

Reassuringly, the errors are "near misses": the model confuses genuinely related categories, not random ones. Place/Time/Quantities all collapse toward Demographics because they're all factual descriptors of the respondent.

Run 4: the key methodological finding

Exact match was measuring the wrong thing

  • Exact match: 18%. Judge ≥ 4: 99%.
  • 94 / 94 questions that did not match the gold string still scored ≥ 4 for meaning.
  • Correlation between exact match and judge: r ≈ 0.04, essentially none.

Takeaway

The low exact-match was a metric artefact, not bad questions. We report judge alignment for the question stage.

This is the headline of Run 4. The 18% exact match looked alarming, but the judge shows 99% of questions are good. The two metrics don't correlate at all (r=0.04): they measure different things. Conclusion: for question quality, trust the judge, not string matching.
Discussion

What it means

  • An off-the-shelf 9B model, with careful prompting and zero training, reaches ~76% concept / ~63% structure on a 22-way expert taxonomy: a strong baseline.
  • Concept identification is largely solvable by prompting + good examples + output constraints.
  • Structure assignment is the genuine frontier: fine distinctions, rare codes, theory-dependent.
  • Question generation is essentially solved: the judge scores it 4.99/5; the 18% exact match was a metric artefact, not a quality problem.
  • Even when the taxonomy label is wrong, assertions still read as faithful paraphrases (judge 4.21 vs 4.61 when correct); taxonomy errors ≠ broken meaning.
  • One catch: the judge is the same model (self-grading) → scores may be optimistic; an external 72B judge is the honest next check.
The interpretation in a few bullets. The split between "concept is easy, structure is hard" is the main scientific takeaway and it directly shapes the next steps.
Discussion

Limitations: being honest

Data

115 rows; many concepts at n = 3 → noisy per-concept estimates. One annotator's scheme.

Metric

Exact-match understates question quality; judge scores exist but are self-graded, with no human or external-judge check yet.

Model & runs

Single model, greedy decoding (temp 0), single run → no variance estimates.

Task ambiguity

Some concept/structure distinctions are inherently debatable, even for experts: a ceiling on achievable accuracy.

Naming the limitations pre-empts the toughest questions. The n=3 issue and the self-grading limitation are the two most important to acknowledge.
Conclusion & Next Steps

Takeaways

  • A prompt-driven pipeline turns raw indicators into theory-grounded survey items end-to-end.
  • Iterative, error-driven prompting added +18pp concept accuracy with no training.
  • Structure codes are the frontier; the LLM-as-judge confirms questions are excellent (4.99/5) despite low exact match.

Next steps

  • External judge (Qwen2.5-72B) to remove self-grade bias
  • Structure-focused few-shot
  • LoRA fine-tuning on the assertion stage
  • Expand & balance the gold set
Wrap up: the pipeline works, prompting got us a long way, structure is the open problem, and the next steps follow directly from the limitations: an external judge, targeted few-shot, light fine-tuning, and more data.
